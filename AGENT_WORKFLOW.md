# Agent 标准工作流

## 单一来源

- `AGENTS.md`：项目硬规则和任务路由器。
- `AGENT_WORKFLOW.md`：session 生命周期和最低验证要求。
- `docs/ai/DOCUMENTATION_CONTRACT.md`：文档自愈规则。
- `docs/ai/MODULES/*.md`：各业务模块的当前状态、边界和专项验证。
- 当前代码、配置、测试和生成脚本：行为事实来源；与文档冲突时先查清代码，再同步文档。

不要在多个文档重复维护同一套完整流程。专项文档只补充模块差异，并链接回本工作流。

## 1. 接收与定位

1. 确认当前仓库、分支、技术栈和用户最新请求。
2. 运行 `git status --short`，识别已有修改、暂存内容和未跟踪文件。
3. 阅读 `docs/OWNER_VISION.md`、`docs/ai/PROJECT_CONTEXT.md`，再按 `AGENTS.md` 路由表读取专项文档。
4. 查找现有实现、调用方、测试、配置和文档，不凭文件名或旧记忆推断当前行为。
5. 同一工作区存在其他任务修改时，明确本任务文件和 hunk 边界，不覆盖、不回退、不顺手提交。

## 2. 任务分级

### 必须先计划

- 完整功能、跨文件修改、重构、删除代码、架构调整、跨项目联调。
- 计划必须说明项目、技术栈、已读文件、需求理解、涉及文件、步骤、风险、验证和回滚边界。
- 用户确认前只做只读分析，不修改业务代码。

### 可以直接处理

- 范围明确的小修复、文案替换、单处样式校正、用户明确要求的命令操作。
- 仍需先读相关文件并检查工作区，不能凭空修改。

## 3. 实现循环

1. 选择符合现有架构的最小改动面。
2. 修改前说明将改哪些文件和为什么。
3. 先完成一个可验证的小闭环，再继续下一段；不要堆到最后一次性验证。
4. 行为代码修改后立即运行最接近该行为的检查或真实页面操作。
5. 发现问题先确认是本次回归、既有问题还是环境问题，再修复；不得用隐藏错误、吞异常或伪造成功状态绕过。
6. 新增公共抽象前确认至少有真实复用点，或它能明显隔离复杂业务边界。

## 4. 真实应用验证

行为代码修改不能只依赖静态阅读或生产构建成功，必须实际运行对应应用或接口。

### 前端 UI

1. 启动或复用当前仓库的 Vite 服务，先确认端口和进程归属，不终止无关服务。
2. 用真实 hash 路由打开修改页面，执行本次改动对应的用户路径。
3. 检查页面错误、控制台错误、失败网络请求和 Vite overlay。
4. 布局或交互涉及响应式时，至少验证一个桌面视口和一个移动视口。
5. Canvas、上传、拖拽、裁剪、导入和导出必须使用有代表性的真实或合成样本，不能只确认控件存在。

### API、服务和数据

1. 启动真实后端或明确的本地 fallback。
2. 请求实际接口并核对成功、输入错误和上游失败分支。
3. 数据生成器必须运行 checker 或验证产物结构，不能只查看脚本源码。

### 允许不启动应用的任务

- 仅修改 Markdown、注释、gitignore 或不影响运行时的项目规则。
- 纯文档任务仍需执行链接/路径核对、`git diff --check` 和相关一致性搜索。
- 如果运行环境不可用，必须记录具体阻塞、已完成的替代验证和残余风险。

本机工具路径和端口覆盖可写入 ignored `docs/ai/LOCAL_ENVIRONMENT.md`。公共文档不得硬编码个人目录。

## 5. 最低验证矩阵

| 改动类型 | 最低验证 |
| --- | --- |
| Markdown / 项目规则 | 路径与引用搜索、`git diff --check` |
| TypeScript 纯逻辑 | `npm run typecheck` 或对应测试 |
| Vue UI / CSS | 类型检查、本地化检查、真实页面桌面验证；涉及响应式时增加移动端 |
| API / 外部导入 | 实际请求成功与错误分支、契约检查 |
| Canvas / 导出 | 真实预览、导出产物尺寸/内容检查、对应回归 |
| 数据生成 | 运行生成器、checker、检查公开产物边界 |
| 发布 | `docs/ai/DEPLOYMENT_CHECKLIST.md` 的完整检查和上线后烟测 |

模块文档若规定了更严格的命令或样本，以模块文档为准。

## 6. 文档自愈

实现过程中出现以下变化时，在同一任务同步文档：

- 路由、页面入口、数据源、API、端口或公开边界改变。
- 模块状态从计划变为已实现，或旧说明不再成立。
- 新增持久化键、导入导出格式、性能边界或关键回归样本。
- 同一问题第二次需要依赖聊天历史才能处理。

更新方法遵守 `docs/ai/DOCUMENTATION_CONTRACT.md`。不要把一次性过程记录追加成永久流水账。

## 7. 收尾审查

1. 运行本任务最低验证和必要的更高层检查。
2. 检查 `git diff --check`、`git status --short` 和最终 diff。
3. 搜索调试代码、临时路径、密钥形态、未授权资产和不应公开的信息。
4. 核对文档是否仍与代码一致。
5. 总结修改文件、原因、行为影响、验证结果和残余风险。

## 8. 提交与推送

1. 只暂存本任务文件或 hunk，检查 `git diff --cached --name-only` 和 staged diff。
2. commit message 和交付说明使用中文并写明范围。
3. 普通增量推送核对待推送 commit 的作者、提交者和 GitHub CLI 账号。
4. 重建仓库、迁移远端、force push、tag 或 release 时，额外审计完整历史和所有 refs。
5. 未经用户要求不自动提交或推送；长任务进入下一阶段前遵守 `AGENTS.md` 的确认规则。
