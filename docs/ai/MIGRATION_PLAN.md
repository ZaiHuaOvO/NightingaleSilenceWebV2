# 迁移计划

## 目的

本文件记录第一阶段将 `NSHome`、`NSPortable`、`NSGlamour` 迁入 V2 的顺序、边界和验证标准，避免 `ARCHITECTURE_PLAN.md` 越写越大。

整站愿景以 `docs/OWNER_VISION.md` 为准：夜莺不语是个人/工具网站，`FFXIV` 只是当前第一阶段分类。迁移原则是先保留旧项目可用，再逐步在 V2 中建立等价能力。旧项目后端在早期作为临时兼容 API、行为契约和回归样本保留；最终后端可以按 V2 的新规则重写。未完成等价验证前，不删除旧项目和旧部署入口。

## 项目边界

| 来源项目 | V2 目标 | 初期边界 |
|---------|---------|---------|
| `NSHome` | `#/` 首页 | 迁入为个人站视觉入口，旧目录可暂时保留备份 |
| V2 新增 | `#/ffxiv` FFXIV 分类导航页 | 只做工具导航，不承接旧项目业务逻辑 |
| `NSPortable` | `#/ffxiv/plate` 铭牌编辑器 | 旧 Node.js 后端先继续独立运行，端口 `3456`，同时作为新后端契约参考 |
| `NSGlamour` | `#/ffxiv/glamour` 幻化工具 | 旧 Flask 后端先继续独立运行，端口 `8765`，同时作为新后端契约参考 |

## 总体顺序

### 阶段 0：V2 地基

目标：

- 确认 Vue、Router、Pinia、TypeScript、Vite proxy 可用。
- 建立公共 CSS、公共组件、公共 composable 的最小骨架。
- 建立模块文档目录：`docs/ai/MODULES/`。
- 落实 `ARCHITECTURE_PLAN.md` 的样式变更分级：全站级、模块/分类级、页面专属。

验证：

- `npx vue-tsc --noEmit`
- `npm run build`
- Vite dev server 可启动。

### 阶段 1：NSHome 首页

目标：

- 迁入首页内容和工具入口。
- 使用 V2 公共 CSS 和公共组件建立统一视觉语言。
- 不引入复杂后端依赖。
- 首页主入口指向 `#/ffxiv`，不直接承载具体工具列表。
- 保留后续承载博客、创作信息和其他分类的扩展余地。

边界：

- 首页不承接 Plate/Glamour 业务逻辑。
- 只提供品牌视觉入口、站点介绍和外链。
- 首页粉蓝像素风、人物舞台、贴纸和浮动图标是页面专属视觉，不写进全站公共组件默认样式。
- 旧 `NSHome` 不立即删除。

验证：

- `#/` 可访问。
- 移动端和桌面端布局稳定。
- 主入口链接指向 `#/ffxiv` 或对应占位路由。

### 阶段 1.5：FFXIV 分类导航页

目标：

- 建立 `#/ffxiv` 分类导航页。
- 展示 `glamour`、`plate`，并为未来 FFXIV 工具预留扩展空间。
- 使用公共按钮、卡片、面板和导航样式。

边界：

- 不承接 `NSPortable` 或 `NSGlamour` 的复杂业务逻辑。
- 不在首页展开工具列表。
- 不继承首页强像素装饰；分类页使用公共组件和模块/分类级布局。

验证：

- `#/ffxiv` 可访问。
- `#/ffxiv/glamour` 和 `#/ffxiv/plate` 入口可点击到正式页或占位页。

### 阶段 2：确认工具迁移优先级

NSHome 完成后，由用户决定先迁移：

- `NSPortable` 铭牌编辑器。
- 或 `NSGlamour` 幻化工具。

不要在未确认优先级时同时大规模迁移两个复杂工具。

### 阶段 3A：NSPortable 迁移

目标：

- 先抽取旧项目 API、素材、预设、Canvas 渲染和导出行为契约。
- 将旧项目核心 Canvas 渲染逻辑或等价新实现迁入 `src/lib/plate/`。
- 将页面交互拆到 `src/pages/plate/`。
- 使用 `/api/plate`、`/img`、`/img-preview` 接入旧后端。
- UI 控件尽量使用公共组件和公共样式。

边界：

- 初期旧 Node.js 后端继续可用；最终后端可以按新规则重写，但必须先通过契约和回归验证。
- 不改变旧素材路径和已验证的导出行为。
- 不为了 Vue 结构调整 Canvas 坐标或渲染参数。

验证：

- 选层、预览、导出 PNG/ZIP/PSD/JSX 流程与旧项目对齐。
- 游戏素材加载路径正确。
- 字体、像素坐标、图片裁剪无明显回归。

### 阶段 3B：NSGlamour 迁移

目标：

- 先抽取旧项目 API、装备解析、染剂规则、模板渲染和导入行为契约。
- 将旧项目模板渲染器或等价新实现迁入 `src/lib/glamour/`。
- 将 `/template` 和 `/equipinfo` 的核心交互拆到 `src/pages/glamour/`。
- 使用 `/api/glamour` 接入旧 Flask 后端。
- 保持装备、染剂、模板、导入、导出规则兼容。

边界：

- 初期旧 Flask 后端继续可用；最终后端可以按新规则重写，但必须先通过契约和回归验证。
- 不改变 `.chara` 解析、装备映射和染剂规则的 API 字段。
- 不删除旧 `NSGlamour` 的 Jinja 页面，直到 V2 形成等价工作流。

验证：

- `.chara` 导入、网页链接导入、文字识别导入可用。
- 装备和染剂选择逻辑与旧项目一致。
- 模板预览、图片裁剪、PNG 导出与旧项目对齐。
- 多语言装备名和染剂名能正确显示。

### 阶段 4：生产部署切换

目标：

- V2 构建产物作为统一前端。
- Nginx 反向代理旧后端 API。
- 旧路径按需要做跳转或兼容。

边界：

- 未通过完整回归前，不下线旧项目。
- 不把内部路径、密钥、调试接口暴露到公开站点。
- 旧项目删除必须另起计划，证明无外部引用和无回滚需求。

验证：

- 生产域名首页可访问。
- `#/ffxiv`、`#/ffxiv/plate` 和 `#/ffxiv/glamour` 可访问。
- API、素材、字体、上传、导出在生产代理下可用。
- 浏览器控制台无路径和 CORS 错误。

## 共享能力沉淀顺序

迁移过程中优先沉淀这些公共能力：

1. `src/styles/`：颜色、字体、间距、按钮、面板、表单、状态提示。
2. `src/components/`：公共按钮、面板、工具栏、输入控件、状态提示、弹窗。
3. `src/composables/`：`useFetch`、窗口/拖拽/Canvas 调度等。
4. `src/lib/`：与 Vue 解耦的业务逻辑和 Canvas 渲染。

页面独有的东西留在页面目录；跨两个页面以上复用的东西应考虑上移。

样式沉淀必须先判断层级：

- 全站级：影响所有页面，才能进入公共 CSS 或公共组件默认样式。
- 模块/分类级：只影响某类页面，留在模块目录或模块样式。
- 页面专属：只影响单页，留在页面 scoped style 或页面私有组件。

## 新后端重写原则

如果决定重写 `NSPortable` 或 `NSGlamour` 后端，应遵守：

1. 不从空白想象接口，先把旧后端行为整理成 API 契约和样本集。
2. V2 对外路径保持 `/api/plate/*` 和 `/api/glamour/*` 稳定，后端实现可以在代理后替换。
3. 新后端可以重新组织目录、类型、校验和服务边界，但不能无计划改变前端可见字段。
4. 数据密集逻辑必须有真实样本回归，尤其是装备/染剂、Canvas 导出、素材路径和多语言。
5. 新旧后端并行验证通过前，旧服务不删除、不下线。

## 不做事项

- 不在早期合并两个旧后端为一个服务。
- 不在没有视觉回归的情况下重写 Canvas 渲染。
- 不把旧项目全部文件机械搬进 V2。
- 不为了“目录好看”改动已经稳定的数据字段。
- 不在未确认部署策略前切换到 clean path。
- 不把第一阶段的 FFXIV 工具边界误写成整站长期边界。
