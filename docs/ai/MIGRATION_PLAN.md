# 迁移计划

## 目的

本文件记录第一阶段将 `NSHome`、旧 `NSPortable`、`NSGlamour` 迁入 V2 的顺序、边界和验证标准，避免 `ARCHITECTURE_PLAN.md` 越写越大。旧 `NSPortable` 在 V2 中的新模块名统一为 `NSPlate`。

整站愿景以 `docs/OWNER_VISION.md` 为准：夜莺不语是个人/工具网站，`FFXIV` 只是当前第一阶段分类。迁移原则是先保留旧项目可用，再逐步在 V2 中建立等价能力。旧项目后端在早期作为临时兼容 API、行为契约和回归样本保留；其中 `NSPlate` 的正式素材/预设数据源已经切到静态 manifest + COS/CDN，旧 `NSPortable` 后端只作为显式 legacy/dev fallback、manifest 生成源和旧接口参考。未完成等价验证前，不删除旧项目和旧部署入口。

## 项目边界

| 来源项目 | V2 目标 | 初期边界 |
|---------|---------|---------|
| `NSHome` | `#/` 首页 | 迁入为个人站视觉入口，旧目录可暂时保留备份 |
| V2 新增 | `#/ffxiv` FFXIV 分类导航页 | 只做工具导航，不承接旧项目业务逻辑 |
| 旧 `NSPortable` / V2 `NSPlate` | `#/ffxiv/plate` 铭牌编辑器 | V2 默认静态 manifest + COS/CDN；旧 Node.js 后端端口 `3456` 仅作 legacy/dev fallback、manifest 生成源和旧导出接口参考 |
| `NSGlamour` | `#/ffxiv/glamour` 幻化工具 | 旧 Flask 后端先继续独立运行，端口 `8765`，同时作为新后端契约参考 |

## 总体顺序

当前开工口径：

- 先聚焦第一阶段两个 FFXIV 工具：`NSPlate` 和 `NSGlamour`。
- 不在这一轮继续推进 `Silence` 角色档案，只保留已有规划文档和路由命名结论。
- 不直接搬完整 UI 或重写后端；先把旧项目接口、样本和可验收的第一段迁移切出来。

## 迁移护栏

旧项目迁入 V2 时，必须先区分四类东西，不允许混在一个页面切片里一起搬：

| 类型 | 可以做什么 | 不能做什么 |
|------|------------|------------|
| 工具页骨架 | 统一顶部导航下方的工作台结构、预览区、配置区、tab/toolbar 容器。 | 不在骨架里塞具体业务字段、旧接口统计、迁移 debug 信息或废弃 UI。 |
| 业务数据契约 | 记录旧 API 字段、样本、旧配置键、素材 ID、Canvas 参数和回归样本。 | 不把未确认字段直接变成用户可见正式 UI。 |
| 迁移实验切片 | 可以临时验证 API、素材路径、预设匹配、字段解析。 | 临时切片必须标记清楚，不得默认为正式工作台结构。 |
| 正式用户界面 | 只展示用户真正需要操作的信息，并走本地化和统一控件体系。 | 不展示为了 AI/开发者核对才存在的统计、旧字段名、内部状态。 |

### 不要踩的坑

1. 不要先把旧项目右栏整块搬进新页面，再事后整理样式。复杂工具应先确定 `FFXIV` 工具页共享骨架，再把 `NSPlate`、`NSGlamour` 的业务 slot 填进去。
2. 不要让一个页面组件同时承担工具页骨架、业务数据请求、旧字段适配、右栏控件样式和 Canvas 预览布局。发现页面组件开始膨胀时，先拆外壳和业务组件。
3. 不要把旧项目里的历史字段、废弃字段、调试字段、兼容字段直接放到用户可见 UI。它们可以进契约文档、字段定义表或迁移样本，但默认不进入正式界面。
4. 不要把“字段盘点”和“正式 UI 设计”混为一谈。字段盘点需要保留旧名和来源；正式 UI 只显示经确认的游戏术语、本地化文案或用户确认文案。
5. 不要一边迁移业务，一边随手发明新样式。样式变更先按 `ARCHITECTURE_PLAN.md` 判断全站级、模块级、页面专属；不能判断时先留局部并标记为临时。
6. 不要半调用全局像素风、半写页面私有控件，造成同一工作台里出现两套视觉语言。工具页骨架和基础控件必须先统一，再做业务细节。
7. 不要为了“看起来有进展”把迁移统计、接口数量、缺失素材、旧配置状态长期留在正式工作台。迁移辅助信息只能放调试面板、开发注释或模块文档。
8. 不要把旧配置里的 `name`、`label`、`title` 等用户可编辑或历史兼容字段直接当作 V2 固定文案。固定文案必须有稳定 key 和本地化策略。
9. 不要在没有回归样本的情况下改 Canvas 坐标、导出尺寸、字体参数、素材裁剪、染剂/装备字段含义。
10. 不要把可复用组件过早上升到全站公共层。先在模块内私有复用；当 `NSPlate` 和 `NSGlamour` 都验证需要同类结构时，再提升到 `src/pages/ffxiv/components/` 或 `src/components/`。

### 迁移前必须提前规划

每个复杂工具进入业务迁移前，先确认：

1. **共享骨架**：`#/ffxiv/glamour` 和 `#/ffxiv/plate` 是否使用同一个工具页工作台外壳；哪些区域由外壳负责，哪些由业务 slot 负责。
2. **可见信息**：用户真正需要看到和编辑的字段清单；迁移辅助信息和旧字段盘点不默认可见。
3. **数据来源**：每个字段来自旧 API、旧配置、本地静态数据、游戏客户端数据，还是用户输入。
4. **术语和本地化**：游戏术语优先来自游戏客户端或已确认来源；AI 生成的临时文案不能作为正式展示文案。
5. **样式层级**：全站控件、FFXIV 工具页骨架、单工具私有控件分别放在哪里，哪些样式允许复用，哪些必须局部隔离。
6. **临时切片退出条件**：每个迁移实验切片完成后，要明确哪些代码会保留，哪些要删除/隐藏/改名，避免临时 UI 变成事实正式页面。
7. **回归样本**：至少准备旧项目真实样本、空数据样本、错误样本；涉及 Canvas/导出/装备/染剂时必须准备视觉或结构回归。
8. **兼容和清理策略**：旧 localStorage、旧字段名、旧分类名、缺失素材和废弃功能是否迁移、忽略、隐藏或另设导入工具。

### 临时迁移切片标记规则

临时迁移切片必须在模块文档中写清：

- 为什么需要这个切片。
- 读取了哪些旧项目来源。
- 哪些信息只用于开发核对，不能进入正式 UI。
- 本切片不迁移哪些能力。
- 后续统一骨架或正式 UI 完成后，哪些临时代码要被替换。

### 阶段 0：V2 地基

目标：

- 确认 Vue、Router、Pinia、TypeScript、Vite proxy 可用。
- 建立公共 CSS、公共组件、公共 composable 的最小骨架。
- 建立模块文档目录：`docs/ai/MODULES/`。
- 落实 `ARCHITECTURE_PLAN.md` 的样式变更分级：全站级、模块/分类级、页面专属。

验证：

- `npx vue-tsc --noEmit`
- `npm run build`
- Vite dev server 可启动。

### 阶段 1：NSHome 首页

目标：

- 迁入首页内容和工具入口。
- 使用 V2 公共 CSS 和公共组件建立统一视觉语言。
- 不引入复杂后端依赖。
- 首页主入口指向 `#/ffxiv`，不直接承载具体工具列表。
- 保留后续承载博客、创作信息和其他分类的扩展余地。

边界：

- 首页不承接 NSPlate/NSGlamour 业务逻辑。
- 只提供品牌视觉入口、站点介绍和外链。
- 首页粉蓝像素风、人物舞台、贴纸和浮动图标是页面专属视觉，不写进全站公共组件默认样式。
- 旧 `NSHome` 不立即删除。

验证：

- `#/` 可访问。
- 移动端和桌面端布局稳定。
- 主入口链接指向 `#/ffxiv` 或对应占位路由。

### 阶段 1.5：FFXIV 分类导航页

目标：

- 建立 `#/ffxiv` 分类导航页。
- 展示 `glamour`、`plate`，并为未来 FFXIV 工具预留扩展空间。
- 使用公共按钮、卡片、面板和导航样式。

边界：

- 不承接 `NSPlate` 或 `NSGlamour` 的复杂业务逻辑。
- 不在首页展开工具列表。
- 不继承首页强像素装饰；分类页使用公共组件和模块/分类级布局。

验证：

- `#/ffxiv` 可访问。
- `#/ffxiv/glamour` 和 `#/ffxiv/plate` 入口可点击到正式页或占位页。

### 阶段 2：确认工具迁移优先级

NSHome 完成后，由用户决定先迁移：

- `NSPlate` 铭牌编辑器。
- 或 `NSGlamour` 幻化工具。

不要在未确认优先级时同时大规模迁移两个复杂工具。

### 阶段 2.5：NSPlate / NSGlamour 共同契约盘点

目标：

- 同时盘清两个旧项目的 API、样本数据、资源路径、旧本地状态和第一段可见迁移目标。
- 验证旧服务能作为临时兼容 API 或回归样本运行：`NSPlate` 旧 Node 服务端口 `3456`，`NSGlamour` 旧 Flask 服务端口 `8765`。
- 补齐 `docs/api/nsplate.md`、`docs/api/nsglamour.md` 中“待补充”的真实字段和样本描述。
- 明确两个工具各自的第一段垂直切片，但只选择一个先进入业务实现。

当前状态：`NSPlate` 已完成该阶段的大部分契约盘点和核心工作台迁移，后续以 `docs/ai/MODULES/nsplate.md` 的剩余计划为准；`NSGlamour` 已完成阶段 0 契约地基，并接入第一段 equipinfo 工作台，后续继续按 `docs/ai/MODULES/nsglamour.md` 推进。

推荐执行顺序：

1. 启动或确认旧服务可用。
2. 通过 V2 代理验证轻量接口：
   - `GET /api/plate/presets`
   - `GET /api/plate/files`
   - `GET /api/glamour/health`
   - `GET /api/glamour/stains`
   - `GET /api/glamour/ui-localization`
3. 为每个接口记录成功响应、错误响应、字段含义和安全边界。
4. 收集最小回归样本：
   - `NSPlate`：一组预设、一组素材列表、一个最小导出 payload、一个超限错误样本。
   - `NSGlamour`：一个私有本地角色配置样本、一个染剂样本、一个装备搜索样本、一个外部链接或文本导入样本。
5. 决定先做哪条业务线的第一段可见迁移。

第一段可见迁移建议：

| 工具 | 建议第一段 | 原因 |
|------|------------|------|
| `NSPlate` | 预设和素材浏览 + 基础画布预览壳 | 更快看到页面进展，但 Canvas 和导出回归风险高。 |
| `NSGlamour` | 染剂/装备基础数据读取 + 导入结果结构展示 | 更早处理数据正确性，适合作为后续模板和装备编辑的地基。 |

边界：

- 不同时实现两个完整编辑器。
- 不在没有样本对照时改 Canvas 坐标、导出尺寸、装备字段或染剂字段。
- 不把旧项目文件机械复制进 V2。
- 不删除旧服务、旧页面或旧部署入口。

验证：

- `npm run build`
- 旧服务轻量接口能通过 V2 代理访问。
- 文档中的接口字段来自真实响应或旧代码，不凭空补字段。
- 第一段实现前，先列出涉及文件、风险点和回归样本。

### 阶段 3A：NSPlate 迁移

目标：

- 先抽取旧项目 API、素材、预设、Canvas 渲染和导出行为契约。
- 将旧项目核心 Canvas 渲染逻辑或等价新实现迁入 `src/lib/plate/`。
- 将页面交互拆到 `src/pages/plate/`。
- 使用静态 manifest + COS/CDN 作为 `NSPlate` 默认素材/预设数据源；旧 `/api/plate`、`/img`、`/img-preview` 只保留为显式 legacy/dev fallback 和旧接口参考。
- UI 控件尽量使用公共组件和公共样式。

边界：

- 初期旧 Node.js 后端继续可用；最终后端可以按新规则重写，但必须先通过契约和回归验证。
- 不改变旧素材路径和已验证的导出行为。
- 不为了 Vue 结构调整 Canvas 坐标或渲染参数。

验证：

- 选层、预览、PNG/JPG 和分层 ZIP 流程与旧项目对齐。
- 游戏素材加载路径正确。
- 字体、像素坐标、图片裁剪无明显回归。

### 阶段 3B：NSGlamour 迁移

目标：

- 先抽取旧项目 API、装备解析、染剂规则、模板渲染和导入行为契约。
- 将旧项目模板渲染器或等价新实现迁入 `src/lib/glamour/`。
- 将 `/template` 和 `/equipinfo` 的核心交互拆到 `src/pages/glamour/`。
- 使用 `/api/glamour` 接入旧 Flask 后端。
- 保持装备、染剂、模板、导入、导出规则兼容。

边界：

- 初期旧 Flask 后端继续可用；最终后端可以按新规则重写，但必须先通过契约和回归验证。
- 不改变本地角色配置解析、装备映射和染剂规则的 API 字段。
- 不删除旧 `NSGlamour` 的 Jinja 页面，直到 V2 形成等价工作流。

验证：

- 隐式本地配置导入、网页链接导入、文字识别导入可用。
- 装备和染剂选择逻辑与旧项目一致。
- 模板预览、图片裁剪、PNG 导出与旧项目对齐。
- 多语言装备名和染剂名能正确显示。

### 阶段 4：生产部署切换

目标：

- V2 构建产物作为统一前端。
- `NSPlate` 的 `/data/plate/*.json` 随 V2 静态文件部署，图片走 COS/CDN。
- 仍需后端的模块由 Nginx 反向代理 API；旧路径按需要做跳转或兼容。

边界：

- 未通过完整回归前，不下线旧项目。
- 不把内部路径、密钥、调试接口暴露到公开站点。
- 旧项目删除必须另起计划，证明无外部引用和无回滚需求。

验证：

- 生产域名首页可访问。
- `#/ffxiv`、`#/ffxiv/plate` 和 `#/ffxiv/glamour` 可访问。
- API、素材、字体、上传、导出在生产代理下可用。
- 浏览器控制台无路径和 CORS 错误。

## 共享能力沉淀顺序

迁移过程中优先沉淀这些公共能力：

1. `src/styles/`：颜色、字体、间距、按钮、面板、表单、状态提示。
2. `src/components/`：公共按钮、面板、工具栏、输入控件、状态提示、弹窗。
3. `src/composables/`：`useFetch`、窗口/拖拽/Canvas 调度等。
4. `src/lib/`：与 Vue 解耦的业务逻辑和 Canvas 渲染。

页面独有的东西留在页面目录；跨两个页面以上复用的东西应考虑上移。

样式沉淀必须先判断层级：

- 全站级：影响所有页面，才能进入公共 CSS 或公共组件默认样式。
- 模块/分类级：只影响某类页面，留在模块目录或模块样式。
- 页面专属：只影响单页，留在页面 scoped style 或页面私有组件。

## 新后端重写原则

如果决定重写 `NSPlate` 或 `NSGlamour` 后端，应遵守：

1. 不从空白想象接口，先把旧后端行为整理成 API 契约和样本集。
2. `NSPlate` 默认对外数据路径为 `/data/plate/*.json` 和 COS/CDN 素材；仍需后端的模块保持 `/api/...` 命名空间稳定，后端实现可以在代理后替换。
3. 新后端可以重新组织目录、类型、校验和服务边界，但不能无计划改变前端可见字段。
4. 数据密集逻辑必须有真实样本回归，尤其是装备/染剂、Canvas 导出、素材路径和多语言。
5. 新旧后端并行验证通过前，旧服务不删除、不下线。

## 不做事项

- 不在早期合并两个旧后端为一个服务。
- 不在没有视觉回归的情况下重写 Canvas 渲染。
- 不把旧项目全部文件机械搬进 V2。
- 不为了“目录好看”改动已经稳定的数据字段。
- 不在未确认部署策略前切换到 clean path。
- 不把第一阶段的 FFXIV 工具边界误写成整站长期边界。
