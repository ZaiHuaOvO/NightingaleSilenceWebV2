# 页面开发指南

## 新增页面前置规则

新增页面或迁移旧项目页面前，必须先建立或更新对应模块文档：

```text
docs/ai/MODULES/<module>.md
```

模块文档至少写清：

1. 页面来源和迁移范围。
2. 目标路由。
3. 页面数据来源、API、后端端口。
4. 组件拆分计划。
5. 公共 CSS、公共组件、公共 composable 的复用策略。
6. 验证方式。

如果页面非常简单，不单独建立模块文档也可以，但必须在计划里说明原因。复杂页面默认必须建模块文档。

## 新增页面流程

1. 阅读 `PROJECT_CONTEXT.md`、`MODULE_MAP.md` 和对应模块文档。
2. 确认路由状态仍为未实现或待迁移，避免重复创建。
3. 在 `src/pages/<module>/` 下创建页面入口。
4. 页面内按需拆分子组件到 `components/` 子目录。
5. 在 `src/router/index.ts` 的 `routes` 数组中添加 lazy import 路由。
6. 需要共享的逻辑放入 `src/composables/` 或 `src/lib/`。
7. 需要共享的 UI 样式优先放入 `src/styles/` 或公共组件。
8. 更新 `MODULE_MAP.md` 和对应模块文档的状态。

## 展示文案规则

新增页面、功能入口、空状态、说明文字、按钮文案、卡片简介、状态标签等会展示给用户看的文案，必须区分来源：

1. 用户已经明确提供或确认的品牌名、项目名、工具名、分类名、路由名、导航名，可以按用户原文使用。
2. 必要技术标识可以保留，例如 `NSGlamour`、`NSPlate`、旧 `NSPortable`、`FFXIV`、API path、端口、文件名。
3. 如果只是为了让功能骨架先跑起来、但用户尚未提供正式文案，统一写 `占位用，待编辑`。
4. AI 不要自行发挥写正式宣传语、功能介绍、卡片简介、空状态说明或 About 文案。
5. 代码中如果需要复用占位文案，优先使用统一文案 key，例如 `src/config/site.ts` 中的 `textKeys.placeholder`，不要散落重复字符串。
6. 后续用户给出正式文案时，再替换对应占位文本。
7. 所有用户可见固定文案、按钮文字、状态文字、aria-label、title、placeholder 和弹窗标题必须写成 `src/locales/` 文案 key；页面和组件中不直接硬编码展示文本。
8. `src/config/site.ts` 只放路由、入口、API、项目稳定信息和文案 key，不放真实展示文案。组件渲染时通过 `useLocale().t(key)` 取当前语言文本。
9. 旧项目迁移时，数据键、图层名、文件分类名、游戏术语等如果承担兼容字段，不能为了本地化直接改原始键；应在显示适配层增加本地化 label。

## 路由注册示例

当前 Router 使用 hash mode：

```ts
// src/router/index.ts
{
  path: '/ffxiv/xxx',
  component: () => import('@/pages/xxx/XxxPage.vue'),
}
```

浏览器访问路径形态为：

```text
http://localhost:5173/#/ffxiv/xxx
```

## 页面组件模式

```vue
<template>
  <main class="xxx-page">
    <!-- 优先组合公共组件和公共 class -->
  </main>
</template>

<script setup lang="ts">
// 页面 orchestration 写在这里
// 复杂数据处理放 composables 或 lib/
// API 调用优先经过 useFetch.ts（创建后）
</script>

<style scoped>
/* 只写页面特有样式 */
</style>
```

## CSS 和组件规则

所有样式修改先按 `ARCHITECTURE_PLAN.md` 的“样式变更分级规则”判断层级：

1. 全站级样式：改公共 CSS 或公共组件，应该影响所有页面。
2. 模块/分类级样式：只影响某一类页面，例如 FFXIV 分类和工具页。
3. 页面专属样式：只影响单页，例如首页人物舞台、贴纸和浮动图标。

判断不清楚时，不要直接改公共样式。

涉及 `NSPlate`、`NSGlamour` 或未来同类工房工作台时，必须同时阅读 `docs/ai/WORKBENCH_STYLE_CONTRACT.md`。工房页允许布局不同，但应遵守同一套工作台控件、密度、状态提示、响应式和视觉降噪规则。

优先级：

1. 先查公共组件是否能满足：按钮、面板、表单、工具栏、状态提示、弹窗、tabs。
2. 再查公共 CSS class 和 CSS 变量是否能满足。
3. 仍然不够时，再写页面级 `<style scoped>`。
4. 如果同类样式在第三个地方出现，考虑沉淀成公共组件或公共 class。

不要：

- 不要让每个页面复制一套按钮、卡片、输入框、工具栏样式。
- 不要把页面专属配色写成新主题。
- 不要为了一个页面局部需求污染全局 class。
- 不要把首页人物舞台、贴纸装饰和强视觉编排写入公共组件默认样式。
- 不要把 `day / night` 写成两套主题；它们只是全站像素风的明暗配色模式。
- 不要在已有本地化 UI 中硬编码仅英文标签。

## API 和数据规则

- 不要在组件里硬编码 `localhost` 或端口。
- `useFetch.ts` 创建前，涉及 API 的页面应先在计划里明确临时调用方式。
- Canvas、解析、导出、映射等复杂逻辑应放 `src/lib/`，不要塞进页面组件。
- 上传和外部链接导入都视为不可信输入。

## 验证清单

- [ ] `npm run dev` 启动无编译错误。
- [ ] 浏览器访问对应 hash 路由页面正常渲染。
- [ ] 页面在 560px / 900px / 1120px+ 宽度下布局正常。
- [ ] TypeScript 类型检查通过：`npx vue-tsc --noEmit` 或 `npm run build`。
- [ ] 无控制台错误或警告。
- [ ] 如果改 Canvas、拖拽、上传、裁剪、导出，使用真实数据和浏览器实际验证。

## 禁止事项

- 不要跨页面直接引用其他页面私有组件。
- 不要在页面组件中写复杂 Canvas 或数据处理逻辑。
- 不要引入新依赖，除非先说明原因、影响、替代方案并获得确认。
- 不要把“计划目录”当成“已存在目录”；动手前必须确认文件真实存在。
