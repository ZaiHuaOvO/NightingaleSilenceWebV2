# 代码结构规则

## 目的

本文件用于避免旧项目中“按业务一路写成一大坨”的问题在 V2 中复发。规则适用于后续 `NSPlate`、`NSGlamour` 和其他复杂工具模块。

核心目标不是追求抽象好看，而是让每个改动能落在清楚的边界内：修 UI 时不碰业务规则，修染剂时不碰页面布局，修导出时不碰导入流程。

## 分层原则

复杂功能必须按以下边界拆分：

```text
页面组件
→ composables
→ services / adapters
→ lib 纯业务逻辑
→ API / 后端
```

### 页面组件

页面组件只负责组织 UI：

- 组合公共组件和页面私有组件。
- 接收用户输入。
- 展示 loading、error、empty、success 状态。
- 调用 composable 或 service。
- 连接路由、标题、基础页面状态。

页面组件不要直接写：

- Canvas 绘制算法。
- 装备/染剂合法性判断。
- 外部链接解析。
- 导出 payload 组装。
- 大型数据归一化。
- 后端原始字段适配。

### Composables

Composable 负责把页面交互组织成可复用状态逻辑：

- 表单状态。
- 弹窗和选中项。
- 请求生命周期。
- 本地草稿状态。
- 拖拽、上传、键盘、窗口尺寸等 UI 行为。

Composable 可以调用 service 和 lib，但不要成为第二个“超大页面文件”。

### Services / Adapters

Service 和 adapter 负责边界适配：

- API 请求。
- 旧后端字段适配为 V2 内部模型。
- 外部链接导入入口。
- 图片、素材、字体等资源路径解析。

规则：

1. 组件不直接依赖旧后端原始字段。
2. service 不写复杂业务规则，只做边界转换和调用编排。
3. 后端路径、端口和 API base 统一来自配置或 `apiBoundaries.ts`。

### Lib 纯逻辑

`src/lib/` 放与 Vue 解耦的核心业务逻辑：

```text
src/lib/
├── plate/
└── glamour/
```

适合放入 `lib` 的逻辑：

- 数据结构归一化。
- 装备、染剂、素材、图层、模板规则。
- Canvas 渲染入口。
- 导出数据收集和校验。
- 文案格式化。
- 可用真实样本做回归验证的算法。

规则：

1. 尽量写成输入明确、输出明确的函数。
2. 不直接访问 DOM。
3. 不直接读取 `localStorage`。
4. 不直接发起网络请求。
5. 不依赖 Vue 组件实例。

## 模块边界

### NSPlate

建议边界：

```text
src/lib/plate/
├── assets/      素材索引、素材路径、预览路径
├── canvas/      画布尺寸、坐标、缩放、基础绘制工具
├── export/      PNG / ZIP / PSD / JSX 导出数据
├── layers/      图层模型、排序、启用状态、校验
├── presets/     预设解析和应用
└── render/      预览和最终渲染
```

不要把素材列表、预设应用、图层面板 UI、导出 payload 全写进一个页面文件。

### NSGlamour

建议边界：

```text
src/lib/glamour/
├── equipment/   部位、候选、主副手、合法性检查
├── dyes/        染剂槽、空染色、不可染色、多语言染剂
├── importers/   .chara、石之家、EC、成段文字导入
├── templates/   模板定义和模板数据适配
└── renderers/   Canvas 渲染器
```

不要把导入解析、装备合法性、模板渲染和文案格式化混在同一个 UI 文件里。

## 文件体积警戒线

这些不是硬性报错，而是需要停下来评估的警戒线：

| 类型          | 警戒线    | 应对                  |
| ------------- | --------- | --------------------- |
| Vue 页面组件  | 约 300 行 | 拆子组件或 composable |
| 普通组件      | 约 250 行 | 拆内部组件或样式      |
| Composable    | 约 250 行 | 拆成更小的状态逻辑    |
| 业务 lib 文件 | 约 500 行 | 按领域拆目录          |
| 单个函数      | 约 80 行  | 拆步骤函数并命名      |

如果超过警戒线但暂时不拆，必须能说明原因，例如迁移中间态、算法不可分割或即将删除。

## 重复逻辑规则

同类逻辑第一次出现可以放在局部。

第二次出现时，允许先保持局部，但要确认是否真的只是巧合。

第三次出现时，应考虑沉淀为：

- 公共组件。
- 公共 CSS class。
- composable。
- `src/lib/` 纯函数。
- service / adapter。

不要为了“未来可能复用”提前抽空壳；也不要让真实重复长期分裂。

## 样式边界

迁移复杂工具时，样式也要按边界落位，不要把旧项目局部 CSS、Style Lab 实验类和正式公共组件混写在同一个页面里。

迁移前先把样式分成三类：

| 类型         | 示例                                                       | 建议位置                                                      |
| ------------ | ---------------------------------------------------------- | ------------------------------------------------------------- |
| 全站视觉语言 | 颜色、字体、像素边框、阴影、焦点态、按钮基础状态           | `src/styles/theme.css`、`src/styles/components.css`、公共组件 |
| 模块通用结构 | NSPlate 工作台左右栏、NSGlamour 模板编辑区、模块内重复面板 | 模块私有组件或模块级样式                                      |
| 页面特殊细节 | 单页画布比例、特殊预览区、只出现一次的局部间距             | 页面 `<style scoped>`                                         |

规则：

1. 正式页优先使用 `AppButton`、`AppField`、`AppToolbar`、`AppTabs`、`AppStatus`、`AppPixelWindow` 等公共组件。
2. 正式页不得直接使用 Style Lab 的 `.ns-pixel-*` 实验类；实验类只允许出现在 `#/style-lab` 或隐藏调试页。
3. 实验 CSS 不从 `src/styles/index.css` 全站入口导入，只能由实验页按路由加载。
4. 修改公共组件默认样式前，先确认影响所有调用方；只服务单页的需求应写 modifier、模块组件或页面 scoped style。
5. 迁移旧功能时，先还原行为和数据边界，再做视觉微调；同一次改动不要同时改业务规则、API 适配和大面积样式。

## 状态分层

状态至少分为三类：

| 状态类型   | 示例                                   | 建议位置                |
| ---------- | -------------------------------------- | ----------------------- |
| 服务端数据 | 素材列表、预设、装备搜索、染剂表       | service + store/cache   |
| 用户草稿   | 当前装备、模板设置、图层配置、上传图片 | store 或模块 composable |
| UI 状态    | 当前 tab、弹窗开关、loading、选中项    | 页面或 composable       |

不要把这三类状态混在一个巨大 store 中。

## 后端结构方向

后端重写时也要避免单文件膨胀：

```text
routes / controllers  处理 HTTP 参数和响应
services              编排业务流程
domain                纯业务规则
repositories          读取 CSV / JSON / 数据库 / 文件
adapters              外部站点、图片代理、旧接口兼容
```

后端规则：

1. Controller 不写复杂业务。
2. Domain 不依赖 HTTP 请求对象。
3. Repository 不决定业务规则。
4. Adapter 负责隔离外部服务变化。
5. 上传、外部链接、图片代理、导出接口都必须有输入限制和错误边界。

## 迁移规则

迁移旧功能前先做三件事：

1. 写或更新对应模块文档。
2. 抽取旧行为契约和真实样本。
3. 明确本次只迁移哪个边界。

迁移中禁止：

- 为了快，把旧大文件整段塞进 Vue 页面。
- 边迁移边改视觉参数，导致无法判断回归来源。
- 让组件直接依赖旧后端原始字段。
- 临时手写数据映射代替正式构建流程。

## 验证要求

涉及复杂业务时，至少准备这些验证：

- TypeScript 构建验证。
- API 输入输出样本。
- 导入/解析真实样本。
- Canvas 或导出视觉样本。
- 错误输入和空数据样本。

没有自动测试时，也要把手工验证样本记录在模块文档或 Review Guide 中。
